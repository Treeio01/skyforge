# Аудит админки — 2026-04-29

Прошёл по всем `MoonShine\Resources` и моделям. Список ниже — что есть и чего не хватает для нормальной операционки.

---

## Глобально (общее, не на отдельных страницах)

- **Дашборд `DashboardPage`** — есть базовый `ValueMetric` блок (всего юзеров, активных, баланса, скинов). Не хватает: сегодняшние/недельные срезы (depo, withdraw, upgrade win-rate), графики по времени (см. эпик A6).
- **Globally нет** интерфейса для бан/разбан юзера ОДНОЙ кнопкой с modal-причиной (есть `is_banned` switcher, но без `ban_reason` и confirmation).
- **Нет экспорта** ни в одном resource'е (CSV/XLSX). Для аудита, бухгалтерии — критично.
- **Нет bulk-actions** (массовое одобрение выводов, бан группы юзеров, активация скинов и т.п.). MoonShine поддерживает — не настроено нигде.
- **Soft delete `User`** — на модели `User` есть `SoftDeletes` (deleted_at), но `UserResource` не показывает удалённых. Восстановление через UI отсутствует.

## Resource по resource

### UserResource
**Есть:** список с пагинацией, фильтры `is_banned`/`is_admin`, switcher'ы.
**Не хватает:**
- Кнопка **«Забанить»** с модалкой ввода `ban_reason` (сейчас приходится открыть форму, переключить switcher, написать причину, сохранить).
- Колонка **«UTM-метка»** (через `utmMark.slug`) для быстрой фильтрации по источнику.
- Фильтры: «Дата регистрации (от-до)», «Баланс (от-до)», «Есть Trade URL да/нет», «utm_source», «utm_mark_id».
- Метрики: всего юзеров, активных за 24ч/7д, забаненных, без trade_url.
- QueryTags (быстрые табы): «Активные», «Забаненные», «Без trade URL», «Только админы», «Удалённые».
- Action **«Сбросить баланс»** / **«Начислить»** (admin-correction с записью в Transaction).
- Action **«Применить chance_modifier»** (для A2 — подкрутка) — поле уже есть в БД.
- Колонка регистрационного IP / референрера для антифрод.

### SkinResource
**Есть:** фильтры по категории и активности.
**Не хватает:**
- Метрики: всего скинов, активных, средняя цена, есть-без image.
- QueryTags: «Активные», «Без цены», «Без редкости» (`rarity is null`).
- Bulk action **«Деактивировать выбранные»** / **«Активировать»**.
- Фильтры: цена (от-до), редкость, есть/нет image_path.

### UpgradeResource
**Есть:** фильтр по `result` (win/lose).
**Не хватает:**
- Метрики: за сегодня (кол-во, сумма ставок, выигрышей, фактический house_edge).
- QueryTags: «За 24ч», «За неделю», «Только победы», «Подозрительные» (chance < 5% и win, например).
- Фильтры: диапазон chance, ставка-сумма, target-сумма, по user_id, по дате.
- Колонка **UTM-метки** через `user.utmMark.slug`.

### DepositResource
**Есть:** только базовый список.
**Не хватает (критично):**
- Фильтры: статус (pending/completed/failed), метод (card/crypto/skins), валюта, диапазон сумм, диапазон дат, по user.
- QueryTags: «Pending», «Completed сегодня», «Failed за 7 дней».
- Метрики: за сегодня (сумма, шт.), за месяц, средний чек, % failed.
- Action **«Пометить как completed»** (ручная коррекция при висящих pending).
- Колонка **UTM-метка** (откуда пришёл депозит).

### WithdrawalResource
**Есть:** базовый список.
**Не хватает (критично):**
- Фильтры: статус (`pending/processing/completed/rejected/failed`), сумма, дата, user.
- QueryTags: «Ожидают обработки», «Завершены за 24ч», «Отклонены».
- Action **«Подтвердить»** / **«Отклонить»** с modal-причиной (rejection_reason).
- Метрики: ожидают обработки шт., средний размер, среднее время обработки.

### TransactionResource
**Есть:** базовый список.
**Не хватает:**
- Фильтры: тип транзакции, направление (debit/credit), user, диапазон сумм/дат.
- QueryTags по типам: «Депозиты», «Выводы», «Покупки маркета», «Продажа скинов», «Апгрейды», «Бонусы».
- Метрики: оборот за день/неделю.
- **Read-only** (никакого create/edit) — этот ledger не должен редактироваться руками.

### PromoCodeResource
**Есть:** colorful index с типами/использованиями/expires.
**Не хватает:**
- Метрики: активных промокодов, всего использований, общий бонус выдан.
- Action **«Скопировать URL с промокодом»** (генерация ref-ссылки).
- Связь с `PromoCodeUsage` (история использования — кто, когда, сколько получил) — отдельный resource или sub-page.

### FaqItemMoonResource / FaqCategoryMoonResource
**Есть:** работает.
**Не хватает:**
- Drag-n-drop сортировка `sort_order` (сейчас числами).
- Bulk activate/deactivate.

### SettingResource
**Есть:** низкоуровневый CRUD по таблице.
**Не хватает (UX):**
- Фильтр по `key` LIKE.
- Группировка по префиксу (`online.*`, `social_*`, `seo_*`).
- Inline-edit values (правка прямо в таблице без открытия формы).

### UtmMarkResource
**Есть:** список с counts (юзеры/депозиты/апгрейды/выводы), форма создания, детальная страница.
**Не хватает:**
- QueryTags: «Активные», «Архив», «Ручные», «Авто из UTM».
- Фильтры: source, campaign, по диапазону регистраций.
- Метрики: всего меток, активных, конверсия (юзеры → депозиты в %).
- Action **«Скопировать ссылку»** — `?ref=slug` готовая URL.
- Колонка **сумма депозитов** в рублях (через `deposits()->sum('amount')`).

## Модели без admin-resource'а

Эти таблицы существуют в БД, но не имеют admin-CRUD:

1. **`PromoCodeUsage`** — лог использования промокодов. Нужен read-only resource для антифрод-аудита.
2. **`UserSkin`** — инвентарь юзеров. Нужен resource для просмотра/коррекции (списать скин у юзера, выдать). Сейчас доступа из админки нет.
3. **`SkinPrice`** — история изменений цен скинов. Полезно для графиков «как менялась цена за месяц». Read-only.
4. **`UpgradeItem`** — детали скинов в ставке апгрейда (когда юзер ставит несколько скинов). Может рендериться как nested на странице `UpgradeDetailPage`.
5. **`ProvablyFairSeed`** — пары seed'ов для верификации. Read-only resource на случай разбора инцидента «у меня украли деньги, проверьте честность».

## Кросс-cutting улучшения

- **Activity Log просмотр.** Установлен `Spatie\Activitylog`, модели логгируются. Resource для просмотра отсутствует — нет «кто что менял».
- **Failed jobs.** Horizon показывает failed jobs, но в основном меню нет shortcut'а на `/admin/horizon`.
- **2FA** для админов не настроена.
- **Логин-история** админов отсутствует (`moonshine_users` не пишет login attempts).
- **Стат-блоки на индекс-страницах** — почти нигде нет (см. метрики выше). MoonShine v3 поддерживает `Metric` компоненты — все ресурсы оставлены пустыми.

## Приоритеты (моя рекомендация)

**Top 3 (критичны для ежедневной работы):**
1. **WithdrawalResource** — actions «Подтвердить/Отклонить» + фильтры по статусу. Без этого выводы вручную не обрабатываются.
2. **DepositResource** — фильтры + QueryTag «Pending» + action «Пометить completed».
3. **UserResource** — кнопка бана с причиной + UTM-колонка.

**Tier 2 (полезно):**
4. Метрики на индексах (Upgrade, Deposit, Withdrawal) — числа за день/неделю.
5. Bulk-actions (PromoCode, Skin, FAQ).
6. Export CSV хотя бы для Transaction/Deposit/Withdrawal.

**Tier 3 (nice-to-have):**
7. Resource для PromoCodeUsage / UserSkin / SkinPrice.
8. Activity Log просмотр.
9. UTM Marks: «Скопировать ссылку», конверсия.
10. Drag-n-drop FAQ sort.
